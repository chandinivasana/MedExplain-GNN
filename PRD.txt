Product Requirements Document (PRD): MedExplain-GNN1. Project OverviewMedExplain-GNN is an explainable medical reasoning engine. It translates unstructured user text into medical symptoms, uses a Graph Neural Network to predict potential diseases, and traverses a knowledge graph to output contextual dietary precautions (foods to avoid based on biological interactions).2. Core Objectives (The "Why")Move beyond "Black Box" AI: Provide clear, graph-based reasoning for every medical and dietary recommendation.Contextual Nutrition: Link predicted diseases to specific preventative measures and food contraindications.Production-Ready Scaling: Architect the system as a distributed, containerized microservice ecosystem to handle asynchronous AI workloads.3. Technical Architecture & StackThis stack is specifically chosen to demonstrate full-stack AI and DevOps proficiency.ComponentTechnologyResponsibilityFrontendNext.js, TypeScript, TailwindUser interface for inputting symptoms and viewing the explainable results.API GatewayFastAPI (Python)High-performance routing of requests to the AI models.NLP EngineBioBERT (Hugging Face)Named Entity Recognition (NER) to extract exact symptoms from raw text.Reasoning EnginePyTorch GeometricGraph Convolutional Network (GCN) to predict diseases from symptom nodes.DatabaseNeo4j & MongoDBNeo4j for the Knowledge Graph (relationships) and MongoDB for storing user session logs.Queue & CacheRedisManaging asynchronous message passing between the frontend and the heavy AI models.DevOpsDocker & KubernetesContainerizing the microservices and orchestrating the cluster (using Minikube locally).4. Key Features & User FlowSymptom Ingestion: The user types a paragraph describing how they feel into the Next.js interface.Entity Extraction: The text is sent via an API call to the FastAPI server, where BioBERT extracts entities (e.g., ["high fever", "joint pain"]).Graph Reasoning: The extracted symptoms are mapped to the Neo4j database. The PyTorch GCN analyzes the surrounding nodes and predicts a disease (e.g., Influenza: 85%).Explainability & Dietary Precautions: The system queries Neo4j for nodes connected to "Influenza" via the [:CONTRAINDICATED] edge.Final Output: The user receives the prediction, a visual graph explaining the symptoms that led to it, and a list of foods to avoid (e.g., "Avoid sugary foods as they impair immune response during Influenza").Project Execution Plan (Roadmap)This plan breaks the project down into four manageable phases.Phase 1: Data Engineering & Graph Construction (Weeks 1-2)Task 1: Source a dataset mapping symptoms to diseases (e.g., from Kaggle).Task 2: Manually map 10-15 common diseases to specific dietary precautions and "foods to avoid" to create the nutritional logic.Task 3: Spin up a local Neo4j instance using Docker.Task 4: Write Python scripts using the neo4j driver to populate the database with Symptom, Disease, and Food nodes, establishing their edges.Phase 2: Deep Learning Model Development (Weeks 2-3)Task 1: Finalize the BioBERT symptom_extractor.py pipeline to ensure high accuracy on messy input text.Task 2: Write the training loop for the gcn_disease_prediction.py script.Task 3: Connect the PyTorch GCN to query its training data directly from the Neo4j graph instead of using dummy tensors.Task 4: Export the trained GCN model weights (.pt file) for inference.Phase 3: Backend & Microservices Integration (Week 4)Task 1: Build the FastAPI application containing two main endpoints: /extract-symptoms and /predict-disease.Task 2: Integrate Redis to handle the queuing if multiple users hit the AI endpoints simultaneously.Task 3: Write the Cypher queries in the backend to fetch the "Explainability" data and "Foods to Avoid" once the GCN makes a prediction.Phase 4: Frontend & DevOps Orchestration (Week 5)Task 1: Build the Next.js frontend to send inputs and display the results cleanly.Task 2: Write Dockerfiles for the Next.js frontend, the FastAPI backend, and the AI service.Task 3: Create Kubernetes YAML manifests (Deployments, Services, ConfigMaps, and resource limits for the PyTorch containers).Task 4: Deploy the entire stack locally using Minikube and test end-to-end.
